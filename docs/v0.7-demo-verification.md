# v0.7 Demo Stack Verification

Tested: 2026-05-16

Host environment:

- Windows host, PowerShell shell.
- Docker Desktop with Docker Compose v2-compatible `docker compose`.
- Browser verification used the Codex in-app browser, which does not expose an
  injected wallet provider.

Result:

- Clean build succeeded from pinned public tags.
- The App Core image cloned `isoniaos/app-core` tag `v0.7.0-alpha.5`, and the
  Dockerfile package-version check passed.
- `contracts-deploy` completed and wrote runtime config plus seed output.
- Seeded orgs were org `1` and org `2`.
- Control Plane migrations completed.
- Control Plane API, indexer, and projection worker started.
- Initial indexing processed 51 events with projection backlog `0` and failed
  projections `0`.
- Capabilities reported serial activation and typed contract batch activation.
- Capabilities reported bootstrap organization finalization support.
- EIP-5792 wallet batching was reported unsupported and was not the default
  path.
- Seeded org `1` initially reported real `not_finalized` finalization state.

App Core read-surface checks:

- `http://localhost:5173`, `/diagnostics`, `/orgs`, `/orgs/1`,
  `/orgs/1/governance`, `/orgs/1/proposals`, `/orgs/1/proposals/1`, and
  `/orgs/1/graph` loaded without blank screens.
- Proposal detail rendered the new breadcrumb navigation with named breadcrumb
  links instead of raw blue slash-separated breadcrumb text.
- Diagnostics and governance structure pages remained readable.
- Browser console checks for the visited routes showed no React runtime errors,
  Chakra warnings, missing icon import errors, or transaction layout errors.

New Organization wizard checks:

- `/orgs/new` rendered the new wizard with `IsoBreadcrumbs` and vertical
  `IsoSteps`.
- The first step had no visible Back button and did show Next.
- Back appeared after the first step.
- Validation was not shown before relevant interaction on the identity step.
- After filling name, metadata URI, and admin address, the review step enabled
  `Create Organization`.
- `Create Organization` opened the transaction modal.
- The modal showed vertical transaction steps, left Close, disabled primary
  write action on the right when no wallet was connected, and no `In progress`
  button text.

Activation wizard checks:

- `/orgs/1/setup` rendered the simplified five-step activation flow:
  `Bodies`, `Roles`, `Mandates`, `Policy routes`, `Finalization`.
- No generic `Review / Finish` step was present.
- The roles action opened a typed GovCore batch transaction modal with copy
  stating that one typed GovCore batch would be submitted.
- The typed batch modal showed the same vertical transaction progress steps,
  left Close, disabled primary write action on the right when no wallet was
  connected, and no `In progress` button text.
- Mandates conceptual copy was present in the wizard flow, and no mandates
  validation was shown before the mandates inputs were reached/interacted with.

Finalization and post-finalization checks:

- The in-app browser did not expose an injected wallet, so App Core wallet
  execution could not be completed.
- A local Hardhat JSON-RPC transaction finalized org `1` from the seeded
  bootstrap admin account to verify contract, event, indexing, and read-model
  behavior.
- The finalization transaction mined in block `56`.
- Control Plane indexed one `OrganizationFinalized` event and projected one new
  raw event, bringing processed raw events to 52 with projection backlog `0`
  and failed projections `0`.
- `GET /v1/orgs/1/finalization` updated to `finalized` with
  `bootstrapAdminMutationsAllowed: false`.
- A direct post-finalization static `createBody` call reverted.
- After finalization, App Core showed org `1` as finalized, removed
  `Setup / Activation` from the organization navigation after reload, and
  rendered App Core 404 behavior for direct `/orgs/1/setup` visits.

Known non-blocking warnings:

- App Core build reported Vite chunk-size warnings for large web3 bundles.
- Contract compilation during seed reported local GovCore code-size warnings.
  This is expected for the local demo/test contract shape and is not a runtime
  preview blocker.

Blocking or upstream issues:

- App Core `v0.7.0-alpha.5` still renders stale `v0.6 local demo` copy on the
  home route. This is in the pinned App Core build, not in demo-stack runtime
  configuration.
- App Core wallet-driven activation/finalization could not be fully verified in
  this environment because no injected wallet provider was available.

Final verdict:

- Demo-stack pin adoption, clean build, deploy, seed, diagnostics,
  capabilities, read models, transaction modal readiness states, typed batch
  activation modal readiness, finalization indexing, and post-finalization
  setup route behavior passed.
- The design-partner preview is blocked on the stale App Core home copy unless
  that upstream App Core issue is accepted for this preview or fixed in a new
  App Core tag.
